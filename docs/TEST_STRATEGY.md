# Test Strategy — Cumbres

Defines the testing philosophy, tooling, what to test at each level, and coverage targets.

---

## Philosophy

- Test **behavior**, not implementation details.
- Prefer tests that catch real bugs over tests that maximize coverage numbers.
- Keep the test suite fast enough that developers run it locally without friction.
- E2E tests cover critical user flows end-to-end; unit/integration tests cover business logic in depth.
- Don't test framework code (Next.js routing, Supabase Auth internals, shadcn/ui components).

---

## Tooling

| Layer | Tool | Config file |
|---|---|---|
| Unit | [Vitest](https://vitest.dev/) | `vitest.config.ts` |
| Integration | Vitest + local Supabase | `vitest.config.ts` |
| E2E | [Playwright](https://playwright.dev/) | `playwright.config.ts` |
| Coverage | Vitest built-in (V8) | `vitest.config.ts` |

**Local Supabase** (Docker, via Supabase CLI) is used for integration and E2E tests. It mirrors the production schema and is seeded with fixtures before E2E runs.

---

## Test Levels

### Unit Tests (Vitest)

**What:** Pure functions and business logic with no I/O dependencies.

**Where:** `src/lib/__tests__/`, `src/actions/__tests__/` (for pure logic extracted from actions)

**Examples:**
- Payment total calculations (deposit + regular − refunds; guarantee excluded)
- Payment progress percentage (capped 0–100%)
- Auto-confirm logic (deposit payments ≥ required deposit)
- Status transition validation (pending ↔ confirmed ↔ cancelled)
- Date/time utilities (Montevideo timezone formatting)
- Price history helpers
- Input sanitization / validation schemas (Zod)

**Not unit-tested:**
- Components (covered by E2E or not tested)
- Server actions with DB calls (covered by integration tests)
- Third-party integrations (mocked or covered by E2E)

---

### Integration Tests (Vitest + local Supabase)

**What:** Server actions, DB queries, and auth middleware — tested against a real local DB.

**Where:** `src/actions/__tests__/integration/`, `src/lib/db/__tests__/`

**Examples:**
- Creating a reservation and verifying DB state
- Adding a payment and verifying auto-confirm triggers
- Soft-deleting a reservation (verify record stays with `deleted_at`)
- Auth middleware: verify unauthenticated requests are rejected
- Auth middleware: verify Staff cannot access Admin-only routes
- Expense creation with receipt upload (local Supabase Storage)
- Google Calendar sync failure is non-blocking (reservation still saves)

**Setup:**
- Each test file runs `supabase db reset` or uses transactions rolled back after each test.
- Fixtures defined in `tests/fixtures/` (seed data for known state).

---

### E2E Tests (Playwright)

**What:** Critical user journeys from browser to DB, running against the full local app.

**Where:** `tests/e2e/`

**Browser:** Chromium only (internal app; no cross-browser requirement).

**Critical flows to cover:**

| Flow | Priority |
|---|---|
| Login with valid credentials | P0 |
| Login with invalid credentials (error shown) | P0 |
| Create a new reservation (all required fields) | P0 |
| Add a deposit payment → reservation auto-confirms | P0 |
| Add a regular payment | P0 |
| Cancel a reservation and reopen it | P0 |
| Add an expense with receipt upload | P0 |
| View dashboard stats | P1 |
| Create a client | P1 |
| Export reservations (Excel download) | P1 |
| Admin: create event type | P1 |
| Admin: create a user | P1 |
| Staff cannot access admin routes | P1 |

**Setup per E2E run:**
1. `supabase start` (idempotent in CI)
2. `supabase db reset` — apply migrations + seed fixtures
3. `npx playwright test`
4. `supabase stop` (in CI teardown)

**Fixtures / seed data** (`tests/fixtures/seed.sql` or `tests/fixtures/seed.ts`):
- 1 Admin user, 1 Staff user (known credentials)
- 2–3 clients
- 2 event types with prices
- 1 confirmed reservation with payments

## Coverage Targets

| Scope | Target |
|---|---|
| `src/lib/` | 80% line coverage |
| `src/actions/` | 80% line coverage |
| `src/components/` | No target (covered by E2E) |
| `src/app/` (routes) | No target (covered by E2E) |

Coverage is generated by `vitest run --coverage` (V8 provider). Reports are uploaded as CI artifacts.

Coverage thresholds are enforced in `vitest.config.ts`:
```ts
coverage: {
  provider: "v8",
  thresholds: {
    lines: 80,
    functions: 80,
  },
  include: ["src/lib/**", "src/actions/**"],
}
```

---

## What Not to Test

- **Next.js routing / layouts** — framework behavior; not our responsibility to test.
- **Supabase Auth internals** — trust the library; test our integration at the middleware level.
- **shadcn/ui components** — third-party; only test our usage via E2E.
- **Recharts rendering** — visual; not tested programmatically.
- **Happy-path CRUD with no business logic** — if there's no logic to break, there's no test to write.

## Running Tests Locally

```bash
# Unit + integration (watch mode)
npm run test

# Unit + integration (single run)
npm run test:run

# Unit + integration with coverage
npm run test:coverage

# E2E (requires local Supabase running)
supabase start
npm run test:e2e

# E2E in headed mode (debugging)
npm run test:e2e -- --headed

# E2E for a specific file
npm run test:e2e -- tests/e2e/reservations.spec.ts
```

---

## CI Test Execution

See `docs/SDLC.md` → CI Workflow Detail for the full job sequence.

In CI, the `e2e-tests` job:
1. Installs Supabase CLI and starts local Supabase (`supabase start --workdir .`)
2. Runs migrations + seeds fixtures
3. Starts the Next.js app in the background (`npm run build && npm start`)
4. Runs `playwright test`
5. Uploads test results and screenshots as artifacts on failure
6. Tears down Supabase (`supabase stop`)

## Test File Conventions

```
src/
  lib/
    __tests__/
      payments.test.ts       # unit
      reservations.test.ts   # unit
  actions/
    __tests__/
      reservations.test.ts   # unit (pure logic)
      integration/
        reservations.test.ts # integration (DB)
tests/
  e2e/
    auth.spec.ts
    reservations.spec.ts
    payments.spec.ts
    expenses.spec.ts
    admin.spec.ts
  fixtures/
    seed.sql                 # or seed.ts
```
